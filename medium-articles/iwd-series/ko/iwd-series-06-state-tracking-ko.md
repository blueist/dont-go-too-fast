# 상태 추적: 팀을 위한 IWD (State Tracking: IWD for Teams)

*IWD Deep Dive 시리즈 — 6편*

---

## 혼자서는 할 수 있다. 팀에서는?

당신은 이제 IWD를 압니다.

- 명사 뒤의 동사를 찾는 법
- 진짜 "I"를 찾는 법
- WHI로 발굴하고 검증하는 법
- 규정도 동사로 바꾸는 법

혼자서는 할 수 있습니다.

하지만 월요일 아침, 팀 미팅에서:

> PM: "이 기능 언제 끝나요?"
> 
> 당신: "잠깐요, 이 요구사항의 진짜 'I'가 누군지 아직..."
> 
> PM: "그냥 스펙대로 해주세요. 고객이 직접 말한 거예요."

**IWD는 머릿속에서 사라집니다.**

---

## 문제: 보이지 않는 것은 관리되지 않는다

요구사항의 상태는 보통 두 가지입니다:

```
[ ] 안 함
[x] 완료
```

하지만 현실은 더 복잡합니다:

```
"고객이 원한다고 들었어요" → 확인했나요?
"PM이 그렇게 말했어요" → 진짜 주인한테 물어봤나요?
"스펙 문서에 있어요" → 그 스펙이 맞나요?
```

**보이지 않으면 질문하지 않습니다.**
**질문하지 않으면 검증하지 않습니다.**
**검증하지 않으면 잘못된 방향으로 달립니다.**

---

## Status vs State: 다른 질문, 다른 답

대부분의 팀은 **Status(진행률)**를 관리합니다:

```
To-do → In-progress → Done
```

이것은 **"작업이 얼마나 되었는가?"**를 말합니다.

하지만 IWD는 **State(신뢰도)**를 관리합니다:

```
[Estimated] → [Verified] → 구현
```

이것은 **"이 요구사항을 얼마나 믿을 수 있는가?"**를 말합니다.

| | Status | State |
|---|--------|-------|
| 질문 | 얼마나 했나? | 믿을 수 있나? |
| 측정 | 진행률 | 신뢰도 |
| 위험 | 안 보임 | **보임** |

**Status가 100%여도 State가 [Estimated]면, 잘못된 방향으로 100% 간 겁니다.**

---

## 해결책: 요구사항에 "State"를 붙여라

코드에는 `// TODO`가 있습니다.
요구사항에도 상태가 필요합니다.

| 상태 | 의미 | 다음 행동 |
|------|------|----------|
| **[Estimated]** | 가설. 검증 안 됨. | WHI 필요. **개발 금지.** |
| **[Verified]** | WHI 통과. 진짜 문제임. | 개발 가능 |
| **[Corrected]** | 폐기/보류. | 없음. 개발 대상 아님. |

## 엔지니어의 '안전한 거부권'

**[Estimated]는 개발 착수 금지입니다.**

이것은 규칙이 아니라 **권리**입니다.

```
PM: "이거 이번 주까지 해주세요."

엔지니어: "이 기능은 아직 [Estimated] 상태입니다.
         WHI 질문에 대한 주인의 답변이 필요합니다.
         검증 없이 코드를 짜는 건 도박입니다."
```

팀 전체가 이 State를 인지할 때, 엔지니어는 **정당하게 거부**할 수 있습니다.

1편에서 느꼈던 분노를 기억하세요:

> "His mistake was his to make. The burden was mine to carry."

**[Estimated] 라벨은 이 비대칭을 바로잡는 안전장치입니다.**

## Case: 실시간 대시보드

팀이 받은 요구사항:

> "대시보드를 실시간으로 만들어주세요."

**상태 없이:**

```
[ ] 실시간 대시보드 → 바로 WebSocket 구현 시작
```

**상태와 함께:**

```
요구사항 A: [Estimated] "실시간 대시보드"
    ↓
WHI: "실시간이 아니면 어떻게 되나요?"
    ↓
답변: "데이터를 믿을 수가 없어요"
    ↓
WHI: "다른 방법으로 신뢰성을 보장하면 실시간이 필요한가요?"
    ↓
답변: "아니요, 정확하기만 하면 돼요"
    ↓
요구사항 A: [Corrected] — "실시간"은 잘못된 요구사항이었음 (폐기)
    ↓
요구사항 B: [Estimated] "신뢰할 수 있는 데이터" (새로 발굴됨)
    ↓
WHI 통과
    ↓
요구사항 B: [Verified] "I wanna DO trust the data I see"
```

**핵심:** 
- A는 [Corrected] → 휴지통
- B는 새 요구사항 → [Estimated] → [Verified]

**[Estimated]가 붙어 있으면, 팀 전체가 "이건 아직 가설"임을 압니다.**

---

## 상태 전이 다이어그램

```
[Estimated] ──WHI 통과──→ [Verified] → Task 생성
     │
     └──WHI 실패/불필요──→ [Corrected] (끝)
```

**규칙:**

1. 모든 요구사항은 **[Estimated]**로 시작
2. **WHI**를 통과하면 **[Verified]**
3. WHI에서 잘못됐거나 불필요하면 **[Corrected]** (개발 안 함)
4. **[Verified]**만 개발 가능

## 팀 워크플로우

### 1. 요구사항 접수 시

```
PM: "고객이 엑셀 내보내기를 원해요"

개발자: 
"[Estimated] 엑셀 내보내기 — 고객 요청 (PM 전달)"
```

→ 출처와 함께 기록

### 2. 백로그 리뷰 시

```
팀 리더: "[Estimated] 항목이 3개 있네요. 
        이번 스프린트 전에 WHI 해야 합니다."

개발자 A: "엑셀 내보내기, 제가 고객한테 WHI 하겠습니다."
개발자 B: "대시보드 건은 제가 PM한테 확인할게요."
```

→ [Estimated]가 보이면 행동이 생깁니다

### 3. WHI 결과 공유

```
개발자 A: "엑셀 내보내기 WHI 결과입니다.
        
        Q: '엑셀이 없으면 어떻게 되나요?'
        A: '보고서 만드는 데 4시간 걸려요'
        
        Q: '다른 방법으로 보고서를 빨리 만들 수 있으면?'
        A: '엑셀 아니어도 돼요'
        
        엑셀 내보내기 → [Corrected] (폐기)
        새 요구사항: 보고서 생성 시간 단축 → [Verified]"
```

→ 과정이 투명해집니다

### 4. 스프린트 계획 시

```
팀 리더: "[Verified] 항목만 스프린트에 넣습니다.
        [Estimated]는 이번 주 WHI 대상입니다."
```

→ 검증된 것만 구현

## 도구: 간단하게 시작

### 옵션 1: 티켓 제목에 태그

```
[Estimated] 엑셀 내보내기 기능
[Verified] 보고서 생성 시간 단축
```

### 옵션 2: 커스텀 필드

```
Jira/Asana에 "IWD Status" 필드 추가:
- Estimated
- Verified
```

### 옵션 3: 스프레드시트

| 요구사항 | State | 출처 | WHI 결과 |
|---------|-------|------|----------|
| 엑셀 내보내기 | Corrected | PM | 불필요 — 보고서 시간이 진짜 문제 |
| 보고서 생성 시간 단축 | Verified | PM (WHI) | 확인됨 |
| 실시간 대시보드 | Corrected | 고객 | 불필요 — 데이터 신뢰가 진짜 문제 |
| 신뢰할 수 있는 데이터 | Verified | 고객 (WHI) | 확인됨 |

## 저항에 대처하기

### "시간 없어요"

```
PM: "WHI 할 시간 없어요. 그냥 하세요."

대응: "10분이면 됩니다. 
      이거 안 하고 3주 만들다가 
      '이거 아닌데' 듣는 거랑 비교하면요."
```

### "고객이 원하는 거예요"

```
PM: "고객이 직접 말한 거예요."

대응: "[Estimated]로 기록하고, 
      제가 고객한테 WHI 한 번만 하겠습니다.
      맞으면 [Verified]로 바꾸면 되죠."
```

### "스펙이 이미 확정됐어요"

```
PM: "계약서에 있어요."

대응: "규정도 동사가 될 수 있어요. (5편 참조)
      계약서의 글자가 아니라 의도를 구현하면
      고객도 더 만족합니다."
```

## 팀 문화로 정착시키기

### Day 1-3: 정리

```
- 기존 요구사항 분류: 합의된 것 → [Verified] / 애매한 것 → [Estimated]
- 새 요구사항 규칙: [Estimated]로 시작
- "이건 아직 가설이에요"라고 말하는 연습
- WHI는 아직 안 해도 됨
```

### Week 1-2: 실천

```
- [Estimated] 항목에 WHI 시작
- WHI 결과 공유
- [Verified] 항목만 스프린트에 포함
```

### Month 2+: 습관

```
- [Estimated] 보면 자동으로 WHI
- 팀원끼리 서로 질문
- "WHI 했어요?" "[Verified] 됐나요?"가 자연스러운 질문이 됨
```

---

## 팀에 이 질문을 추가하라

**"[Verified] 됐나요?"**

이 질문 하나가:
- 진짜 'I'를 확인하고 (Part 3)
- 동사를 확인하고 (Part 2)
- WHI를 확인합니다 (Part 4)

개발 시작 전, 이 질문 하나면 충분합니다.

---

## 선언

1. **보이지 않는 것은 관리되지 않습니다.** 요구사항의 상태를 명시하세요.

2. **[Estimated]는 부끄러운 게 아닙니다.** 모르는 걸 모른다고 인정하는 것이 프로페셔널입니다.

3. **[Verified]만 구현하세요.** 가설을 구현하는 것은 도박입니다.

4. **상태 추적은 팀의 생존 매뉴얼입니다.** 혼자 잘해도 팀이 방향을 잃으면 소용없습니다.

**"가설에는 라벨을 붙여라. 그래야 검증할 수 있다."**
