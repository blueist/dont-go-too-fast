# SNP: 작은 걸음도 가치 있다

## 왜 우리는 계단이 아니라 길 위에 있는가

---

![image](https://i.ibb.co/chsvxnV2/07.png)

*이 글은 "압박 속에서 품질을 유지하는 소프트웨어 개발 철학" 시리즈의 일부입니다.*

---

**테스트가 필요하다는 건 안다.**

당신의 시스템에 충분한 테스트가 있는가?

---

테스트를 만들기로 했다.

먼저 뭘 테스트할지 설계했다. 로그인 기능 하나에 정상 케이스, 에러 케이스, 경계값, 동시성... 설계하고 보니 필요한 테스트가 수십 개였다.

몇 개를 만들어봤다. 생각보다 어려웠다. Mock은 어떻게 만들지? 비동기는 어떻게 테스트하지? 데이터베이스는? 이 테스트가 제대로 된 건지 어떻게 알지?

그리고 의문이 들었다. 이걸 다 만들어도 충분한 건가? 내가 놓친 케이스는 없나? 테스트가 있어도 버그가 나오면 어떡하지?

시간을 계산해봤다. 일주일? 이주일? 기능 개발도 해야 하는데.

머릿속에 목소리가 들렸다.

*"어차피 다 못 만들 거잖아."*
*"불완전한 테스트가 무슨 의미가 있어?"*
*"차라리 그 시간에 기능을 더 만들어."*

결국 이렇게 말했다.

"지금은 테스트에 쓸 시간이 없어."

테스트는 다음으로 미뤄졌다. 그리고 다음은 오지 않았다.

---

왜 이렇게 되는가?

**"다 못 만들면 의미 없다"고 생각하기 때문이다.**

---

## 두 개발자 이야기

**개발자 A:**

```
테스트 커버리지 80% 달성
    ↓
버그 발견
    ↓
수정 (테스트 추가 안 함)
    "테스트는 이미 충분해. 80%잖아."
    ↓
또 버그 발견
    ↓
수정 (테스트 추가 안 함)
    ↓
또 버그...
    ↓
Chaos
```

**개발자 B:**

```
테스트 커버리지 80% 달성
    ↓
버그 발견
    ↓
"버그가 나왔다 = 테스트가 충분하지 않았다"
    ↓
버그 수정 + 테스트 추가
    ↓
점점 안정
```

**같은 80%에서 시작했다. 결과가 다르다.**

차이가 뭔가?

**개발자 A:** 80% = Pass (목표 달성) → 끝

**개발자 B:** 80% = 여기까지 왔다 → "더 갈 수 있나?" → 계속

같은 숫자를 다르게 본다.

이것이 SNP다.

---

## SNP: Spectrum, Not Pass/fail

SNP는 세상을 보는 눈이다.

```
Pass/Fail 눈:
    허들 넘음 = 성공 (끝)
    허들 못 넘음 = 실패 (무가치)

SNP 눈:
    허들 넘음 = 여기까지 왔다 (계속)
    허들 못 넘음 = 하지만 이전보다 나아졌다 (가치 있음)
```

**Pass/Fail은 계단이다.** 올랐거나, 안 올랐거나. 중간은 없다.

**SNP는 길이다.** 어디에 있든 길 위에 있다. 앞으로 갈 수 있다.

---

## DGTF와 SNP

버그를 발견했다. 함수를 추가해서 고쳤다. 테스트도 통과했다.

**DGTF:**

"잠깐, 내가 지금 맞게 가고 있나?"

멈추고 본다.

**SNP:**

테스트는 Pass다. 하지만...
- 이 함수가 여기 있는 게 맞나?
- 이상한 의존성이 생긴 건 아닌가?
- 더 나은 방법이 있나?

---

**DGTF는 멈추고 묻게 한다.** "내가 지금 맞게 가고 있나?"

**SNP는 현재 상태에서 가이드라인을 준다.** "Pass다. 그리고 더 나을 수 있다."

---

**DGTF without SNP:**

```
버그 발견 → 함수 추가 → "잠깐" → 테스트 Pass → OK, 끝
```

Pass에서 멈춘다. 더 이상 묻지 않는다.

**DGTF with SNP:**

```
버그 발견 → 함수 추가 → "잠깐" → 테스트 Pass... 
    → 근데 이상한 의존성이 생겼네
    → 함수 제거 → 책임 재정의 → 정리 → 테스트 Pass
```

Pass 후에도 묻는다. 더 나은 방향을 찾는다.

**DGTF가 SNP를 품어야 완전하다.**

## 이미 알고 있다

여우와 신 포도.

점프했다. 못 닿았다.
또 점프했다. 못 닿았다.
"저 포도는 어차피 셨을 거야."
떠났다.

**여우: "못 닿으면 의미 없어."**

---

토끼와 거북이.

토끼는 빨랐다. 앞서 나갔다.
"이 정도면 충분해."
낮잠을 잤다.
거북이가 지나갔다.
토끼가 졌다.

**토끼: "여기까지 왔으면 됐어."**

---

어린 시절 우리는 알았다.

여우가 틀렸다는 걸.
토끼가 틀렸다는 걸.
거북이가 맞았다는 걸.

---

그런데.

"테스트 다 못 만들면 의미 없어." ← 여우
"커버리지 80% 달성했으니 됐어." ← 토끼

우리는 거북이를 응원하면서
여우처럼, 토끼처럼 살고 있다.

---

SNP는 새로운 게 아니다.

어린 시절 이미 배웠다.
거북이처럼 하라고.

느려도 멈추지 않으면 된다.
한 걸음도 가치가 있다.

다만 내 일에 적용하지 않았을 뿐이다.

---

## 왜 계속 가야 하는가

거북이는 왜 멈추지 않았을까?

단순히 "이기려고"가 아니다.

---

오늘 만든 테스트 하나.
오늘 발견한 이상한 의존성 하나.
오늘 정리한 책임 하나.

작아 보인다.

하지만 이 작은 것들이 쌓이면,
내일 코드를 바꿀 때 두렵지 않게 된다.

---

좋은 소프트웨어는 "완벽한 소프트웨어"가 아니다.
**"바꿀 수 있는 소프트웨어"**다.

오늘의 작은 걸음이
내일의 교체 포인트(Replaceable Point)가 된다.

완벽한 시스템은 없다.
하지만 어제보다 더 "바꾸기 쉬운" 시스템은 있다.

거북이가 계속 간 이유다.
우리가 계속 가야 할 이유다.

---

## SNP와 번아웃

"계속 가라"가 번아웃을 만드는 거 아닌가?

아니다. **SNP는 Task가 아니다. 마인드셋이다.**

**Pass/Fail 마인드셋:** "허들 못 넘으면 실패" → 불안 → 압박 → 번아웃

**SNP 마인드셋:** "허들 전에도 가치 있다" → 숨 쉴 수 있다 → 다시 갈 수 있다

**SNP는 번아웃의 원인이 아니라, 번아웃에서 보호한다.**

---

## 내일부터

테스트를 작성할 때, 코드 리뷰를 받을 때, 프로젝트를 진행할 때:

**"Pass인가 Fail인가?"만 묻지 말고**

**"여기까지 왔다. 더 갈 여지가 있나?"도 물어라.**

구체적으로:

- 버그를 고쳤다 → "테스트 추가할까?"
- 기능 완성했다 → "더 개선할 부분 있나?"
- 커버리지 목표 달성했다 → "더 추가할 여지 있나?"

작은 질문 하나가 여우와 거북이를 가른다.

*허들을 넘지 못해도 끝이 아니다.*
*우리는 길 위에 있다.*
